01

Product overview

The why

TripTools is a free, browser-based collection of travel utilities organized by travel phase — the SkoolTools model, applied to vacationers. No accounts required. No tracking. Everything runs locally or on free public APIs.

"Give every traveler the tools they need at their fingertips, for free, without noise or distraction."

1.1Mission & north star

  • Mission. Give every traveler — from the first-time cruiser to the seasoned road-tripper — a single, beautifully designed destination that puts every tool they need at their fingertips, for free.
  • North star metric. A traveler should be able to open TripTools the night before a trip, find any tool within 10 seconds, complete their task without friction, and return on every future trip.
  • Breadth + depth. 65+ tools covering 10 trip types. Each tool is purpose-built — as complex as it needs to be, no more.
  • Mobile first. Travelers are on their phones. Every tool must be fully functional on a 375px viewport.

1.2Core principles

  • Privacy first. No login. No account. No data leaves the browser unless a tool explicitly requires it (live currency rates, weather).
  • Offline-aware. Every tool is clearly labeled online or offline-capable. Offline tools work at 30,000 feet.
  • Tool first, brand second. The traveler came here to solve a thing. The brand sets the tone but never the obstacle.
  • Playful, never childish. Travel is exciting. The app should feel that way — without tipping into gimmick.

1.3Target audience

Primary traveler types

International vacationers · Domestic road trippers · Cruise passengers · Theme park visitors · Business travelers · Adventure travelers · Family trip planners · Solo travelers · Group coordinators

Anti-audience

TripTools is not a booking engine, a travel agency, or a social network. We do not recommend hotels, sell flights, or show ads. We are deliberately a personal toolkit for the traveler themselves.

Technical architecture

How it's built

TripTools is a zero-dependency, zero-server, zero-account product. The hub and every tool are standalone HTML files. Nothing is compiled. Nothing is deployed to a backend. Everything runs in the browser.

File structure

triptools/
├── index.html              ← Main hub
├── mytrip.html             ← MyTrip profile
├── assets/
│   ├── playology-logo.svg
│   └── base.css            ← Shared tokens
├── tools/
│   ├── discovery/
│   ├── documents/
│   ├── packing/
│   ├── money/
│   ├── destination/
│   ├── road-trip/
│   ├── cruise/
│   ├── theme-park/
│   ├── business/
│   └── home/
└── design-document.html    ← This file

Data flow

Each tool file is fully self-contained. Shared state (the MyTrip profile) is stored in localStorage under a triptools_* namespace, accessible from any tool on the same origin.

Tools requiring live data call free public APIs directly from the browser using fetch(). No proxy. No auth token on the client for v1.

Offline tools use bundled static data (JSON objects embedded in the HTML) for all reference lookups — country codes, plug types, emergency numbers, airline bag fees, phrasebook content.

3.1Technology decisions

ConcernDecisionRationale
FrameworkVanilla HTML/CSS/JS — no frameworkZero build step. Any file opens directly in Chrome without a server.
StylingCSS custom propertiesConsistent theming across files. No Tailwind/Bootstrap dependencies.
StatelocalStoragePersistent across sessions, no server needed, 5MB limit sufficient.
APIsFree public REST APIsCurrency: exchangerate-api · Weather: Open-Meteo (no key) · UV: Open-Meteo
Print/Exportwindow.print() + CSS @media printEvery tool that produces a list or plan has a print stylesheet.
MapsLeaflet.js + OpenStreetMapFree, no API key. Used only for Road Trip Route Planner.
ChartsChart.js (CDN)Lightweight. Excellent mobile rendering. Budget Planner + Trip Cost Recap.
Public API strategy. All APIs used in v1 must be free-tier with no required account for basic use, or must offer a generous anonymous quota. If an API requires a key, the tool must degrade gracefully with a prompt rather than breaking silently. Rate-limited tools display a clear message when the limit is hit.

4.3Search & filter system

Search behavior

  • Instant results as user types — no submit button. Searches tool name, description, tags, and phase simultaneously.
  • Tag system includes trip-type keywords: "international", "cruise", "road trip", "business", "family". Results highlight matched text.
  • Zero results state shows "Suggest a tool" prompt. Search persists across phase filter changes.

Filter system

  • Phase pills filter by travel phase (10 phases + "All"). A toggle filters by connectivity.
  • Active filters shown as dismissible chips below search bar. Filter state persists in URL query params for shareable links.
  • Coming Soon tools shown as muted cards (not hidden) so users know what's coming.

05

MyTrip profile

Your context layer

MyTrip is a persistent trip context layer stored in localStorage. It ties together data from multiple tools under a single trip identity. Users can create multiple trips and switch between them.

"Set up your trip once. Every tool that can use that context — weather, currency, packing list, jet lag — just knows."

5.1MyTrip data model

localStorage key: "triptools_trips" → JSON Array of Trip objects

Trip {
  id:           string (UUID)
  name:         string          // "Paris Anniversary Trip"
  destination:  string          // "Paris, France"
  departDate:   string (ISO)    // "2025-06-12"
  returnDate:   string (ISO)    // "2025-06-22"
  tripType:     string[]        // ["international", "leisure"]
  homeCurrency: string          // "USD"
  homeAirport:  string          // "SFO"
  travelers:    number          // 2
  notes:        string          // freeform notes
  createdAt:    string (ISO)
  updatedAt:    string (ISO)
}

localStorage key: "triptools_active_trip"      → Trip ID string
localStorage key: "triptools_budget_{tripId}"  → Budget data
localStorage key: "triptools_packing_{tripId}" → Packing list
localStorage key: "triptools_expenses_{tripId}"→ Expense log
localStorage key: "triptools_journal_{tripId}" → Trip journal

5.2MyTrip features

Profile setup

  • Trip name, destination, departure & return date pickers, number of travelers.
  • Home currency selector (150+ currencies) and home airport (IATA code lookup).
  • Trip type tags (multi-select): Business, Cruise, Road Trip, International, Domestic, Adventure, Family, Solo, Group.

Trip management

  • Multiple trips (up to 10 in localStorage). Switch active trip from any tool page.
  • Duplicate an existing trip for repeat destinations. Delete with confirmation.
  • Export trip data as JSON (full backup). Import from JSON. Factory reset.

5.3Context injection

Each tool that benefits from MyTrip context checks for an active trip on load. If found, relevant fields are pre-populated. A banner reads: "Using context from: Paris Anniversary Trip ↗" — tapping it opens MyTrip for editing. Users can always override pre-filled values within the tool.

Connectivity specification

Online vs. offline
"57 of 75 tools work at 30,000 feet. No wifi? No problem — most of TripTools comes with you."

🌐 Online tools (8 tools)

  • Currency Converter — live exchange rates via exchangerate-api
  • Weather Forecast — Open-Meteo API (no key required)
  • Weather-Based Wardrobe Planner — Open-Meteo 14-day forecast
  • UV Index & Sun Safety — Open-Meteo UV data
  • Road Trip Route Planner — Leaflet + OpenStreetMap tiles
  • Visa Requirement Guide — live confirmation link to official sources

✈️ Bundled offline data

  • 200+ destination dataset — climate, costs, best seasons
  • 100+ country visa & passport rules — 6-month rule matrix
  • 30+ airline baggage fee tables — carry-on and checked
  • 15 plug/adapter types — by country, visual reference
  • 100+ country emergency numbers — police, fire, ambulance
  • 25-language phrasebook — 1000+ phrases with pronunciation
  • 12 cruise lines dataset — fees, amenities, gratuities
  • IANA timezone database — full, bundled
Offline-first UX rule. Offline tools must NEVER show a spinner or error when offline — they must respond instantly. Online tools must display their connectivity requirement BEFORE the user tries to use them, not after a failed API call. Every online tool has an offline fallback state that is clearly communicated.
08

Shared UI components

The building blocks

All tools share a common component library for visual consistency. Each tool file includes these components inline — no external CSS dependency required.

8.1Tool card (hub grid)

  • Phase color accent bar along the left edge (4px, phase color). Phase emoji icon and phase name label (12px, uppercase mono).
  • Tool name in Fraunces 500, 16px. One-line description in Inter Tight 400, 13px. Connectivity badge bottom-left.
  • Coming Soon overlay for unreleased tools: 50% opacity, "COMING SOON" badge top-right.
  • Hover: card lifts (translate -1px -1px), 3px solid ink shadow — matching the SkoolTools tool card hover.

8.2Input system

  • Text inputs: full-width, paper-2 background, ink border (1px), ink border on focus, 14px Inter Tight.
  • Select dropdowns: styled to match text inputs, custom arrow. Searchable where list exceeds 20 items.
  • Minimum tap target: 44×44px on all interactive elements on mobile.

8.3Results / output display

  • Results section delineated by a top border and paper-2 background. Key results in Fraunces 500, 24–32px.
  • Supporting detail in Inter Tight 400, 14px, ink-2 color. Charts use Chart.js with paper-toned theme.
  • Copy-to-clipboard on any text result. Animate in: results fade + slide up on calculation (200ms).

8.4Checklist component

  • Custom checkbox: paper background, accent-colored checkmark on check. Checked items: strikethrough, reduced opacity.
  • Progress bar: X of Y items checked, accent-color fill. Category grouping with collapsible sections.
  • State saved automatically to localStorage on every interaction. Add custom items inline.

8.5Print stylesheet (@media print)

  • All tools include a @media print block. Navigation hidden. Backgrounds converted to white. Text to black.
  • Playology + TripTools header injected at top of print output. Tool name and date printed at top.
  • Charts: print-safe colors (darker lines on white background).

Testing & QA standards

Launch readiness
"A tool is not ready until it works on a 375px screen with no internet, prints cleanly, and passes keyboard navigation."

Functional testing checklist

  • All calculations produce correct output for known inputs
  • Edge cases handled: zero, empty, and maximum values
  • localStorage save/load works across page reloads
  • MyTrip context loads and populates fields correctly
  • Online tools handle: no internet, slow internet, API error, rate limit
  • Print view renders correctly in Chrome Print Preview
  • Export functions produce valid CSV / JSON output
  • All internal links navigate correctly

Responsive & accessibility checklist

  • Tested at 375px, 390px, 768px, 1280px, 1920px
  • No horizontal scroll on any viewport under 375px
  • All tap targets ≥ 44×44px on mobile
  • All form inputs have associated <label> elements
  • Color contrast ≥ 4.5:1 for all body text (WCAG AA)
  • Color is never the sole differentiator
  • All interactive elements reachable by keyboard Tab
  • Calculation results announced via aria-live="polite"

11.1Launch readiness definition

A tool is launch-ready when: (1) all functional tests pass, (2) renders correctly at 375px and 1280px, (3) passes keyboard navigation, (4) print view tested, (5) verified in Chrome on macOS and iOS.

11.2Data accuracy standards

  • Last-verified date visible to the user on all bundled static data. Static data reviewed and updated quarterly.
  • Disclaimer on regulatory tools (vaccination requirements, visa rules, driving rules): "Verify with official sources before travel. Rules change frequently."
  • Data version number embedded in each file. Hub footer displays the global data version.

11.3Cross-browser targets

  • Primary: Chrome desktop + mobile — must be pixel-perfect.
  • Secondary: Safari iOS + macOS — must be fully functional.
  • Tertiary: Firefox — functional; minor visual variance acceptable. No IE support required.
